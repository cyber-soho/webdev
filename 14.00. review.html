The Web Server

A.1 What a web server actually is

A web server is two things at the same time: the physical or virtual machine that hosts the files of a website, and the piece of software running on that machine that listens on a network port, reads incoming HTTP (HyperText Transfer Protocol) messages, and writes outgoing ones. When the conversation in industry says "the server is down", almost always it is the second meaning — the daemon, not the hardware.

The mental model that sticks best: the server is a kitchen that never closes. Browsers are customers; HTTP requests are order tickets; HTTP responses are plates of food. The whole back-end exists to read tickets, cook, and ship plates without burning the place down.

A.2 HTTP — the language

HTTP, drafted at CERN (European Organization for Nuclear Research / Conseil Européen pour la Recherche Nucléaire) by Sir Tim Berners-Lee in 1991 and now maintained by the IETF (Internet Engineering Task Force) under RFC 9110, is a plain-text request/response protocol. Every conversation has two halves.

The request carries a method (GET, POST, PUT, PATCH, DELETE), a path, a version, headers, and an optional body. The response carries a status code, headers, and a body. The status codes you must know cold are:

  • 200 OK — everything went fine
  • 201 Created — a new resource was created (typically after a POST)
  • 301 Moved Permanently — the URL (Uniform Resource Locator) changed
  • 400 Bad Request — the client sent something the server cannot parse
  • 401 Unauthorised — the client must log in first
  • 403 Forbidden — the client is logged in but not allowed in
  • 404 Not Found — the resource does not exist
  • 500 Internal Server Error — the server crashed mid-processing

HTTPS (HyperText Transfer Protocol Secure) is the same protocol wrapped in TLS (Transport Layer Security). Identical grammar, encrypted pipe.

A.3 Static vs dynamic content

Static content is whatever already exists as a file on disk and is served byte-for-byte: HTML (HyperText Markup Language), CSS (Cascading Style Sheets), JavaScript files, images, fonts, PDFs (Portable Document Format). The server is just a file clerk.

Dynamic content is whatever is generated for the request. The server runs code — PHP (PHP Hypertext Preprocessor), Python, Ruby, Node.js, Java, C# (C-Sharp) — assembles the response from a database, and ships it. Static is fast and dumb. Dynamic is slower but personalised.

A real server, like the one you saw in Lesson 3 of the Web Server lecture, can play both roles simultaneously, picking which one based on the URL.

A.4 The Big Three server programmes

  • Apache HTTP Server — born in 1995 at the Apache Software Foundation (ASF). Process- or thread-per-request model. Reliable workhorse, highly configurable through .htaccess files. Heavy under massive load.
  • Nginx (pronounced "Engine-X") — born in 2004, written by Igor Sysoev to solve the C10K problem (handling ten thousand simultaneous connections on one machine). Event-driven, asynchronous. Frequently used as a reverse proxy and load balancer.
  • Node.js — born in 2009. Not a traditional web server; it is a JavaScript runtime built on Chrome's V8 engine, maintained by the OpenJS Foundation. Event-driven and non-blocking, making it brilliant for chat apps, streaming, and APIs (Application Programming Interfaces).

Restaurant analogy: Apache is the steakhouse where one waiter serves one table; Nginx is the conveyor-belt sushi bar where one chef feeds hundreds; Node.js is the food truck you build yourself.

A.5 IP addresses, ports and localhost

An IP (Internet Protocol) address is the building's address. A port (a number between 0 and 65535) is the apartment number inside that building. The combined host:port is what a network connection actually targets. Well-known ports worth memorising:

  • 20, 21 — FTP (File Transfer Protocol)
  • 22 — SSH (Secure Shell)
  • 25 — SMTP (Simple Mail Transfer Protocol)
  • 53 — DNS (Domain Name System)
  • 80 — HTTP (unencrypted web)
  • 443 — HTTPS (encrypted web)
  • 3000 — common Node.js development port
  • 3306 — MySQL
  • 5432 — PostgreSQL
  • 27017 — MongoDB

The official registry lives at the IANA (Internet Assigned Numbers Authority).

localhost is the hostname that always resolves to your own machine, IP 127.0.0.1. http://localhost:3000 is "talk to a server running on this very laptop on apartment 3000".

A.6 Routing

Routing decides which piece of code runs based on the URL path and the HTTP method. Without routing, every URL on the same server returns the same answer. With routing, /, /about, /contact, and /api/menu each map to a different handler.

In raw Node.js you write if/else if blocks against req.url and req.method. In Express you write app.get('/path', handler). Same idea, less boilerplate.

A.7 Middleware — the airport metaphor

Middleware is a function that sits between the raw request and the route handler. Each middleware can read the request, modify it, end it early, or pass control to the next link in the chain. It is the airport metaphor in code:

  • Logger middleware — records every request
  • Authentication middleware — checks tokens, blocks unauthorised users
  • Body-parser middleware — converts JSON (JavaScript Object Notation) bytes into objects you can use
  • CORS (Cross-Origin Resource Sharing) middleware — controls which front-end origins can call your API

In Express the contract is (req, res, next) => { ... next(); }. Forget to call next() and the request hangs forever. That is the single most common Express bug junior developers ship.

Week 10 · Module B

Back-End: Node.js, Express, Vue

B.1 Node.js — what it actually is

Before 2009, JavaScript lived only inside the browser. Ryan Dahl took Chrome's V8 engine, pulled it out of the browser, gave it the ability to read files and open sockets, and called the result Node.js — a free, open-source, cross-platform JavaScript runtime. The single biggest consequence: a company can hire one team that speaks JavaScript on both ends of the stack, instead of paying for two languages and a translator between them.

Node.js comes with NPM (Node Package Manager), the world's largest software registry. Over 2.1 million packages are published at npmjs.com. When you type npm install express, that is where the bytes come from.

The single concept that makes Node.js feel different is the event loop: a single-thread loop that hands long-running operations (file reads, database queries, network calls) off to the operating system, registers a callback — a function that will be invoked when the work is done — and immediately moves on to the next request. The barista metaphor is exact: while the espresso machine drips, the barista takes the next order. Heavy companies running on Node.js include Netflix, LinkedIn, PayPal, NASA (National Aeronautics and Space Administration), and Uber.

B.2 Express — the minimalist framework

Express.js is a thin layer of convenience on top of Node's built-in http module. It gives you:

  • Routing — app.get, app.post, app.put, app.patch, app.delete
  • Middleware — app.use(...) to register a middleware function
  • Helpers — res.json(...), res.send(...), res.status(...)
  • URL parameters — app.get('/students/:id', ...), then read req.params.id

A typical Express skeleton:

const express = require('express');
const app = express();
app.use(express.json());                 // parse JSON request bodies
app.get('/students', (req, res) => res.json(students));
app.post('/students', (req, res) => { /* create */ res.status(201).json(newStudent); });
app.listen(3000);

Where Express stops being magical: it does not give you a database, it does not give you authentication, it does not give you templating. You add those yourself. That is the minimalist in "minimalist framework", and it is the reason developers love it.

B.3 Vue — the reactive front-end

Vue.js (pronounced "view") is a progressive JavaScript framework for user interfaces, created by Evan You in 2014. Progressive means you can sprinkle it onto a single page or build a full SPA (Single-Page Application). Companies using Vue include Alibaba, GitLab, and Nintendo.

The four ideas you must have in your head:

  • Components — self-contained reusable bricks, often in a Single-File Component (SFC) .vue file with <template>, <script>, and <style> sections.
  • Reactivity — declare a piece of state with ref(...) or reactive(...); the moment it changes, Vue automatically updates only the parts of the DOM (Document Object Model) that depended on it. You never call getElementById yourself.
  • Directives — special HTML attributes prefixed with v-. Memorise these: - v-if / v-else — conditional rendering - v-for — list rendering, always with a :key - v-bind (shorthand :) — bind an attribute dynamically - v-on (shorthand @) — bind an event listener - v-model — two-way binding for form inputs
  • Two-way data binding — typing into an input updates the variable, and changing the variable updates the input. Achieved with v-model.

A taste of the Composition API:

import { createApp, ref } from 'vue';
createApp({
  setup() {
    const count = ref(0);
    function increment() { count.value++; }
    return { count, increment };
  }
}).mount('#app');

B.4 The full stack picture

   FRONT-END  (browser)              BACK-END  (server)
   -----------------------           --------------------
   Vue.js                            Node.js  (runtime)
   Components, reactivity            Event loop, non-blocking I/O
   v-model, v-for                    Express  (framework)
            │                        Routing, middleware, REST API
            │      HTTP Request                │
            ├────────────────────────────────►│
            │      JSON Response              │

When somebody says "MEVN stack" — that is MongoDB + Express + Vue + Node.js. MERN swaps Vue for React. MEAN swaps Vue for Angular. Same plumbing, different front-end paint.

Databases & Big Data

C.1 The one-line definition

A database is a place that remembers. What changed across sixty years is how it remembers and how much it can remember while many people read and write at the same time.

The four reasons we use a database instead of a flat file or a spreadsheet:

  • Persistence — survives a power cut.
  • Concurrency — many users at once without trampling each other.
  • Integrity — refuses to accept bad data because you told it the rules.
  • Performance — finds one row out of a billion in milliseconds, thanks to indexes.

C.2 A short timeline

  • 1950s — magnetic tape, sequential access only.
  • 1960s — IBM's IMS (Information Management System) for NASA's Apollo programme — hierarchical, tree-shaped data.
  • 1970 — Edgar F. Codd publishes the relational model: everything is a table. The earthquake.
  • 1986 — SQL (Structured Query Language) standardised by ANSI (American National Standards Institute) and ISO (International Organization for Standardization) under ISO/IEC 9075.
  • 1980s–1990s — golden age of RDBMS (Relational Database Management Systems): Oracle, Microsoft SQL Server, IBM Db2. MySQL (1995) and PostgreSQL (1996) bring open-source.
  • 2000s — Google, Amazon, Facebook hit a wall. Birth of NoSQL: MongoDB, Cassandra, Redis, Couchbase.
  • 2010s–2020s — cloud-managed databases (AWS RDS, Azure SQL, Google Cloud SQL) and AI-native vector stores: Pinecone, Weaviate, pgvector.

C.3 Flat files — the ancestor

A flat file is a plain text file where each row is a record and each column is separated by an agreed character. CSV (Comma-Separated Values) is the most common; TSV (Tab-Separated Values) and JSONL (JSON Lines) follow close behind. No engine, no SQL, just bytes on disk. The Esperanto of data — every language and every spreadsheet program agrees on it.

Three rules of thumb:

  • Read-mostly, single-user, fits in memory → flat file is often the right answer.
  • Two writers at once → stop, you will corrupt the file.
  • Indexed lookups across millions of rows → stop, you will be miserable.

And one rule that catches every junior: never parse CSV with split(','). Use the standard library — Python's csv module, Node's csv-parse, Java's Apache Commons CSV. Quoted commas and embedded newlines will silently destroy your day.

C.4 SQL — the relational world

SQL has five sub-languages that you will see on the exam:

  • DDL (Data Definition Language) — CREATE, ALTER, DROP. Defines structure.
  • DML (Data Manipulation Language) — INSERT, UPDATE, DELETE. Changes data.
  • DQL (Data Query Language) — SELECT. Reads data.
  • DCL (Data Control Language) — GRANT, REVOKE. Manages permissions.
  • TCL (Transaction Control Language) — BEGIN, COMMIT, ROLLBACK. Groups statements.

The four sacred letters: ACID.

  • A — Atomicity. All or nothing.
  • C — Consistency. Every transaction moves the database from one valid state to another.
  • I — Isolation. Concurrent transactions do not see each other's half-finished work.
  • D — Durability. Once committed, even a power loss will not erase it.

Relationships happen via keys: every row has a primary key; another table refers to that row by storing the same value in a foreign key column. The engine refuses to create a foreign key pointing at a non-existent row. That is where referential integrity comes from.

A canonical three-table example — Student, Course, Enrolment — appears in the Databases lecture. Memorise the shape: Student has its own primary key; Course has its own; Enrolment has a composite primary key made of (student_id, course_code) plus two foreign keys.

C.5 NoSQL — the four families

NoSQL stands for Not Only SQL (the original "No SQL" was a marketing accident). Four families, each for a different shape of data:

  • Document store — record is a JSON-like document, every record can have a different shape. King: MongoDB.
  • Key-value store — give it a key, get a value, that is all. King: Redis, almost always deployed as a cache.
  • Wide-column store — millions of optional columns per row, sparse storage. Kings: Apache Cassandra, Apache HBase, Google Bigtable.
  • Graph database — nodes and edges, perfect for friendships, fraud rings, recommendations. King: Neo4j.

The unifying trade-off is the CAP theorem, stated in 2000 by Professor Eric Brewer). Given a distributed database, pick at most two of three:

  • C — Consistency. Every read sees the latest write.
  • A — Availability. Every request gets a response, even if not the freshest.
  • P — Partition tolerance. The system keeps working when the network splits.

Network partitions are a fact, so the real choice is C vs A. Banks pick C. Social networks pick A.

The NoSQL counterweight to ACID is BASE:

  • Basically Available
  • Soft state
  • Eventually consistent

Polite shorthand for "we will get there, just not instantly".

C.6 Hybrid, polyglot persistence, and Big Data

Real systems combine multiple databases — polyglot persistence. Spotify might run SQL for accounts, Redis for sessions, MongoDB or Elasticsearch for the music catalogue, a graph or vector engine for recommendations, and a data lake on AWS S3 for raw event logs.

Big Data is defined by the 5 V's:

  • Volume — terabytes, petabytes, exabytes.
  • Velocity — millions of events per second.
  • Variety — text, images, audio, geo, graph.
  • Veracity — can we trust this data?
  • Value — does it produce revenue, research, or a better decision?

The model that defined the field is MapReduce, published by Google in 2004 and open-sourced as Apache Hadoop. Two phases:

  • Map — runs in parallel on each chunk of input, emits (key, value) pairs.
  • Reduce — groups all values sharing a key and combines them.

The modern successor is Apache Spark, which keeps data in memory and runs the same shape much faster. For event streams the standard is Apache Kafka. The landing zones are data warehouses (Snowflake, BigQuery, Redshift) and data lakes (S3, Azure Data Lake, Google Cloud Storage). Databricks coined "lakehouse" to describe a hybrid of the two.

Week 12 · Module D

Application Types & Architectures

D.1 The three components of any web app

Every web application — without exception — has three families of components:

| Component | Role | Restaurant analogy | | --- | --- | --- | | Client (front-end) | What the user sees and clicks | The waiter | | Server (back-end) | The brain that processes logic | The chef | | Database | Where the data lives | The fridge |

The three talk over HTTP/HTTPS, JSON, and a query language. That is it. Every gigantic site — Amazon, Netflix, Meta — is, at its heart, this loop running at scale.

D.2 Tier-based architectures

A tier is a separate layer:

  • 1-tier — UI, logic, and data on a single machine. Microsoft Excel, an offline calculator.
  • 2-tier — client talks directly to the database. Old desktop registration apps. Security nightmare: anyone who decompiles the client sees the database password.
  • 3-tier — client → application server → database. Almost every modern web app. The application server hides the database credentials.
  • N-tier — many specialised layers (CDN, load balancer, app server, cache, DB, backups). Netflix-class systems.

CDN (Content Delivery Network) = a global cluster of servers that deliver static content from the location closest to the user.

Why move from 2-tier to 3-tier? Security and separation of concerns. The client never sees the database password; the application server validates everything before touching storage.

D.3 Monolith vs Microservices

  • Monolith — one giant codebase, every feature deployed together. Hypermarket Carrefour: one building, one cash register system. Pros: simple. Cons: redeploy everything for any change; one bug crashes everything.
  • Microservices — many small independent services, each with one responsibility, each deployable separately. Shopping mall: independent shops sharing a roof. Pros: independent deployment, fault isolation, language freedom. Cons: network complexity, harder debugging, requires DevOps maturity.

Default to a monolith for a small team and a small product. Move to microservices only when the team and the load demand it.

D.4 SPA vs MPA, SSR vs CSR

  • MPA — Multi-Page Application. Each click loads a brand new page from the server. Wikipedia. Server renders the HTML.
  • SPA — Single-Page Application. The page loads once; JavaScript swaps content in and out. Gmail, Trello, Discord web.
  • SSR — Server-Side Rendering. Server builds the final HTML and ships it complete.
  • CSR — Client-Side Rendering. Server sends a near-empty shell; JavaScript builds the page in the browser.
  • PWA — Progressive Web App. A web app that behaves like a native app — installable, offline-capable, push notifications. Two ingredients: a Web App Manifest (JSON file describing the app) per the W3C spec, and a Service Worker (JavaScript file that runs in the background to cache files and handle notifications).
  • JAMstack = JavaScript + APIs + Markup, modern static-first architecture served via a CDN.

Modern frameworks like Next.js, Nuxt, and SvelteKit do both SSR and CSR depending on the route — that is Universal or Hybrid Rendering.

D.5 Mobile applications — five types

| Type | What it is | Tech | Best for | | --- | --- | --- | --- | | Native | One OS at a time, max performance | Swift on iOS, Kotlin on Android | Games, AR, camera-heavy apps | | Mobile Web | A regular website opened in the phone's browser | HTML, CSS, JS | Lowest commitment | | Hybrid | Web app wrapped in a native shell | Apache Cordova, Ionic | Internal tools, prototypes | | Cross-Platform | One codebase compiled for both stores | React Native, Flutter | Most production startups | | PWA | Web that behaves like native | Service Workers + Manifest | Lean startups, weak-network markets |

OS = Operating System. iOS = Apple's mobile operating system. NFC = Near-Field Communication. The two stores you publish to are the Apple App Store and Google Play.

The decision tree: if you need top performance, go native. If you must ship to both stores at the same time on a small team, go cross-platform (Flutter or React Native). If you want zero installation friction and the weakest networks possible, go PWA.

Week 13 · Module E

Security & Testing

E.1 Security is a posture, not a feature

Security is never yes-or-no. It is a posture — a way of looking at your own code and asking "what happens if the user is not who I expect?" The phrase to memorise: the attacker only needs to find one bug; the defender has to find all of them. This asymmetry is why we layer protections (Defence in Depth).

The most important non-profit in this field is OWASP (Open Worldwide Application Security Project). The OWASP Top 10 is the closest thing to a universal starting point for web security. Bookmark it. In Europe, the public body is ENISA (European Union Agency for Cybersecurity); in Spain, INCIBE (Instituto Nacional de Ciberseguridad). The U.S. equivalent for digital identity standards is NIST (National Institute of Standards and Technology), publisher of Special Publication 800-63B on passwords and authentication.

E.2 The OWASP Top 10 — the four you will probably introduce yourself

You do not need to memorise all ten. You need to recognise these four because, statistically, you will introduce them yourself in the next two years:

  • A01 — Broken Access Control. A regular user changes the URL from /profile to /admin/edit and the application happily lets them edit.
  • A02 — Cryptographic Failures. Credit card data sent over plain HTTP, passwords stored in plaintext, session cookies without Secure and HttpOnly flags.
  • A03 — Injection. Especially SQL Injection. The classic.
  • A07 — Identification and Authentication Failures. Weak passwords, no rate limiting, password reset links that never expire.

A fifth that hits everyone at least once is A05 — Security Misconfiguration. Default passwords, exposed admin pages, debug endpoints in production. The lesson is not "don't use defaults" — it is know your defaults. Read what your framework does out of the box.

E.3 SQL Injection — the canonical bug

A web application takes user input and stitches it directly into a database query. The user, knowing this, types not a name but a fragment of SQL. The database executes both. Game over.

query = "SELECT * FROM users WHERE username = '" + input + "' AND password = '" + pw + "'"

If the user types admin' -- as username, the query becomes SELECT * FROM users WHERE username = 'admin' --' AND password = ..., the -- starts an SQL comment, the password check disappears, and the attacker is logged in as admin without knowing the password.

The defence is parameterised queries (also called prepared statements). The query template and the values travel separately; the database driver — not your code — does the escaping. In Python's sqlite3:

cursor.execute("SELECT * FROM users WHERE username = ? AND password = ?", (username, password))

The single difference between safe and unsafe code is the placeholder. Form the habit. Never concatenate user input into SQL — not even "just this once for a quick demo".

E.4 Cross-Site Scripting (XSS)

XSS is what happens when an attacker injects JavaScript into a page and your application delivers that JavaScript to other users' browsers. Three flavours:

  • Stored XSS — the script lives in the database and is served to every visitor. The worst.
  • Reflected XSS — the script comes from a crafted URL the victim clicks. The most common.
  • DOM-based XSS — the script never reaches the server; it is constructed in the browser by client-side code. The trickiest.

What can the attacker do? Read cookies (unless marked HttpOnly), post as the user, log keystrokes, redirect to a fake login, drain a wallet. Anything the page can do.

The two-layer defence:

  • Escape output. Convert <, >, ", ' into &lt;, &gt;, &quot;, &#x27;. Modern templating engines do this by default. Bugs almost always appear when somebody turns escaping off on purpose.
  • Set a Content Security Policy (CSP) — an HTTPS response header that tells the browser to load JavaScript only from approved sources and refuse inline scripts. The reference is the MDN Web Docs CSP page.

In code: prefer element.textContent = userInput to element.innerHTML = userInput. The first treats the string as literal text; the second parses it as HTML.

E.5 Authentication, passwords, sessions

Authentication = proving who you are. Authorisation = deciding what you are allowed to do once you are authenticated. Confusing the two is the most common rookie mistake.

Three rules for passwords, in order of importance:

  • Never store passwords in plaintext. When the database leaks, plaintext passwords are immediately useful to the attacker.
  • Never store passwords with a plain hash either. Algorithms like MD5 (Message Digest 5) and SHA-1 (Secure Hash Algorithm 1) are too fast — a modern GPU (Graphics Processing Unit) cracks them in hours. Use a deliberately slow, salted, memory-hard algorithm: bcrypt, scrypt, or Argon2. Argon2 won the Password Hashing Competition in 2015 and is the current first choice; bcrypt is the safe everywhere-default.
  • Never roll your own. Use the library.

After the password is verified, the user receives a session — usually a long random string in a cookie. That cookie must be marked:

  • HttpOnly — JavaScript cannot read it (protects against XSS-based session theft)
  • Secure — only travels over HTTPS
  • SameSite=Lax or SameSite=Strict — other websites cannot trick the browser into sending it (protects against CSRF / Cross-Site Request Forgery)

Beyond passwords, offer MFA (Multi-Factor Authentication), often called 2FA (Two-Factor Authentication). The factors are something you know (password), something you have (phone, hardware key), something you are (fingerprint, face).

E.6 HTTPS and transport security

HTTPS = HTTP wrapped in TLS (Transport Layer Security), the modern successor to SSL (Secure Sockets Layer). A site running HTTPS gets:

  • Encryption — eavesdroppers see noise.
  • Integrity — tampering is detected.
  • Authentication — the certificate proves the server is who the URL says it is.

Free certificates from Let's Encrypt since 2016 mean nobody has a financial excuse for plain HTTP any more.

Three response headers that cost nothing and stop a lot of attacks:

  • Strict-Transport-Security (HSTS) — tells the browser "always use HTTPS for this domain for the next N seconds".
  • Content-Security-Policy — already discussed, blocks XSS.
  • X-Content-Type-Options: nosniff — prevents the browser from second-guessing the Content-Type header.

E.7 Why testing exists

Tests answer the question: does this still work after other people have changed it? The bug that destroys teams is the feature that worked Monday and silently broke Wednesday because somebody changed an unrelated file. Tests are how teams stop hating each other.

Three philosophies:

  • TDD (Test-Driven Development) — write the test first, watch it fail, write the code that makes it pass.
  • BDD (Behaviour-Driven Development) — extends TDD with English-like syntax so non-developers can read.
  • Test-After Development — the default of most students; works but produces less testable design.

CI/CD (Continuous Integration / Continuous Deployment) runs your tests automatically on every push and, if everything passes, deploys the code to production. CI/CD is impossible without tests; tests are useless without CI/CD.

E.8 The Testing Pyramid

A picture is the lesson:

         /\
        /E2E\        few, slow, brittle, expensive
       /----\
      / Int. \       some, medium speed, medium cost
     /--------\
    /   Unit   \    many, fast, cheap, the foundation
   /------------\
  • Unit tests — one function in isolation, no database, no network. Hundreds of them. Run in milliseconds. Tools: pytest (Python), Jest and Vitest (JavaScript), JUnit (Java), xUnit (C#).
  • Integration tests — two or more units cooperating. Real or near-real database. Dozens of them. Tools: Supertest, Postman, Hurl.
  • End-to-End (E2E) tests — drive a real browser through a real flow. A handful, only for critical paths. Tools: Cypress, Playwright, Selenium.

The mistake every team makes once: inverting the pyramid. Lots of E2E tests, few unit tests. Result: a forty-minute test suite that fails twice a day for unrelated reasons and gets ignored.

The pattern inside any unit test is AAA — Arrange, Act, Assert. Arrange the inputs, act on the function, assert on the result.

E.9 Mobile testing — the Madrid metro tunnel test

A mobile app must be tested on the device class users actually have. Build a device matrix of three to five physical devices that cover ~80% of your users — for Spain in 2026 that means a recent iPhone, a mid-range Samsung Galaxy, a budget Xiaomi, an older iPhone still on security updates.

The four mobile-only concerns to bake into tests:

  • Connectivity — losing signal in a metro tunnel. Does the app queue actions or crash?
  • Permissions — what happens if the user denies camera, location, notifications?
  • Background/foreground — does the session resurrect after switching to WhatsApp and back?
  • Battery & performance — does the app drain battery faster than competitors?

Tools: Appium (cross-platform, WebDriver-based), Maestro (newer, simpler), Detox (React Native), Apple's XCUITest, Google's Espresso. Firebase Test Lab runs your app on a fleet of real devices in the cloud.

E.10 The two ideas to take with you

  • Assume hostility. Every input might be malicious; every default might be wrong.
  • Assume change. Today's working code will be changed tomorrow by you, your colleague, or the version of you who has slept four hours. Tests are how the code defends itself against that change.

TapaBox: the food-delivery start-up that lost its breakfast

Modules touched: A, B, C, E

Scenario. TapaBox is a Madrid food-delivery start-up running a Node.js + Express back-end behind Nginx, a MongoDB cluster on Atlas, and a Vue.js single-page front-end. One Saturday morning the on-call engineer is paged: customer logins are failing, the home page renders but never loads restaurants nearby, and the Stripe payment webhook endpoint is returning 500 errors. The engineer's first instinct is to restart Node — which makes things briefly better, then worse. Logs show a flood of requests to /api/restaurants?lat=...&lng=... from a handful of internet protocol addresses, each request slightly different.

Tasks. (a) Identify what's most likely happening. (b) Describe a triage order. (c) Recommend at least four longer-term fixes that span the stack.

Model answer.

(a) The pattern — many requests, slightly varied query strings, payment endpoint failing under load — is a classic application-layer denial-of-service against an expensive endpoint. The "slightly different" parameters defeat any naive cache. Restarting Node masks the symptom while the queue refills. Stripe's webhooks are timing out because the event loop is saturated by the geo-query traffic, not because Stripe is broken.

(b) Triage order: (1) stop the bleeding at the edge — push a rate-limit rule into Nginx (or Cloudflare if it sits in front) keyed on internet protocol address and on the /api/restaurants path, before touching the application; (2) protect the payment path — webhooks must not share an event loop with public traffic, so route them to a separate process pool or a different upstream so Stripe's retries don't pile up; (3) confirm data integrity — MongoDB shouldn't be corrupted, but check for unindexed scans on the geo-query collection that may explain why each request is so expensive; (4) only now restart anything, and only with the rate limit already in place.

(c) Longer-term: rate-limit at the reverse proxy as a first line of defence, not in application code; add a 2dsphere index in MongoDB so geo-queries are bounded in cost (an unindexed $near query is the kind of thing that brings down a small cluster); separate the public application programming interface from the webhook receiver into different services or at least different processes, because their failure modes are different; cache the restaurants-nearby response at a short interval (even 30 seconds is enough to absorb a flood) using a key like a rounded latitude/longitude pair; introduce structured logging and a request identifier propagated end-to-end so the next time this happens the engineer doesn't have to grep four log files; finally, run a chaos drill quarterly so the on-call book is exercised before the real incident.

The deeper lesson, hammered through Modules A and E, is that Node's single event loop is fast but unforgiving. One slow handler can starve every other request, including the ones that earn the company money.

The hotel chain with the unlocked back door

Modules touched: C, E (heavy on E.6 Misconfiguration)

Scenario. A mid-sized European hotel chain operates an Elasticsearch cluster to power its booking-search feature. A security researcher contacts the chain on a Friday afternoon: the cluster is reachable on the public internet, with no authentication, on the default port. It contains roughly four million reservation records — guest names, addresses, loyalty-programme numbers, partial card data, special-request notes that include passport numbers in some cases. The chain has four hours before the researcher publishes.

Tasks. (a) Map the failure to the OWASP Top Ten. (b) Describe the immediate response, in order. (c) Identify the regulatory exposure under European Union law and what notification deadlines apply. (d) Propose architectural changes so this can't recur.

Model answer.

(a) The primary failure is A05 — Security Misconfiguration: a database left listening on a public interface with default credentials (or none) and no network-level firewall. Secondary failures touch A01 — Broken Access Control (no authentication at all is the most broken access control there is) and A09 — Security Logging and Monitoring Failures (the cluster has been exposed for who-knows-how-long without alerting). If the partial card data is reconstructable, A02 — Cryptographic Failures also applies.

(b) Immediate response: (1) close the network exposure by moving the cluster behind a virtual private cloud or restricting the security group to back-end internet protocol addresses only — do this before anything else, because every minute of exposure widens the breach; (2) rotate any credentials that touched the cluster and assume any data in it is compromised; (3) preserve logs (Elasticsearch's own access logs, the cloud provider's flow logs, the load balancer's logs) for forensic timeline reconstruction; (4) determine the scope — what data was in the index, what queries hit it from non-internal internet protocol addresses; (5) engage legal and the data protection officer.

(c) Under the General Data Protection Regulation, a personal data breach "likely to result in a risk to the rights and freedoms of natural persons" must be notified to the supervisory authority — in Spain that's the Agencia Española de Protección de Datos — within 72 hours of becoming aware of the breach. Where the risk is "high" (passports, financial data both fall into this), the data subjects themselves must also be notified without undue delay. Sector regulators may impose additional duties.

(d) Architectural changes: never expose a database directly to the internet — put it on a private subnet, accessed only by application servers; require authentication and transport-layer encryption on every internal service, not just the public ones (the "castle and moat" model is dead, and zero-trust assumes the inside is hostile too); add automated configuration scanning that fails the deployment if a service binds to a public interface; classify data and stop storing what isn't needed (passport numbers in free-text "special request" notes is a process problem, not a technical one); add access-pattern alerting so any non-application source hitting the cluster pages someone immediately.

The football blog that got an unexpected visitor

Modules touched: E.4 (Cross-site scripting), A (web server), B (Express)

Scenario. A community blog about Real Madrid runs on a small Express stack. Comments accept basic formatting and are stored in PostgreSQL. Six months in, regular readers start reporting their accounts behaving oddly — posting comments they didn't write, following users they don't know. Investigation shows a comment posted weeks earlier on a heavily trafficked match thread containing the literal text:

<img src=x onerror="fetch('/api/comments',{method:'POST',headers:{'Content-Type':'application/json'},body:JSON.stringify({thread:42,body:'visit my-shady-site.example'})})">

Every reader who loaded that thread had their browser fire that request, authenticated with their session cookie.

Tasks. (a) Classify the attack. (b) Explain the chain — input, storage, render, victim. (c) Describe two layers of defence that would have prevented or contained it.

Model answer.

(a) This is stored cross-site scripting (also called persistent XSS). The payload was written once into the database via a comment form, then served to every subsequent visitor of that thread, where it executed in the victim's browser with the victim's session.

(b) The chain: the comment form accepted hypertext markup language without escaping; the back-end stored the raw string in PostgreSQL; the rendering layer (a server-side template, or a Vue component using v-html, or a string concatenation into the document object model) inserted that string into the page without escaping. The browser then parsed <img src=x onerror="...">, failed to load the image (deliberately), fired the onerror handler, and ran fetch as the logged-in user. The cookie went along for the ride because the browser attaches cookies automatically.

(c) Two layers (you can name more, but two is the minimum): (i) escape on output — the rendering layer should never insert user content as raw hypertext markup language. In Vue, {{ comment.body }} (text interpolation) is safe; v-html is dangerous. In Express templates, prefer engines that escape by default. If formatting is needed, parse to a controlled subset using a library like DOMPurify, allow-list specific tags, and never trust attribute values. (ii) Content-Security-Policy header — set a Content-Security-Policy that disallows inline scripts and unsafe-eval, and restricts which origins can be loaded. Even if a payload sneaks through escaping, the browser refuses to execute it. A third defence worth mentioning is HttpOnly and SameSite=Lax (or stricter) on the session cookie, which doesn't stop the XSS but reduces what the attacker can do with it.

The systemic lesson from Module E: trusting input is the original sin, and output encoding is the redemption.

The price-comparison site that gave away free milk

Modules touched: E (testing), B (Express), C (databases)

Scenario. A Madrid price-comparison site scrapes Mercadona, Carrefour, and Lidl prices nightly and shows the cheapest option per product. One morning users report that a litre of milk shows as costing zero euros at every supermarket. The product page goes viral. The comparison site issues an apology and points out, correctly, that no actual sale took place — but reputational damage is real.

The post-mortem reveals: an upstream scraper occasionally returns a price string like "€0.95" with a non-breaking space; a cleaning function strips non-numeric characters with a regular expression that, after a recent change, also stripped the digit "9" in some locales; the unit test for the cleaning function used "1.50" and "0.99" as inputs and passed.

Tasks. (a) Classify the missing test types. (b) Rewrite the test plan for that single function. (c) Explain why this kind of bug is the one the testing pyramid is supposed to catch.

Model answer.

(a) The existing test was a unit test, but with a thin and unrepresentative input set. There's no property-based test (one that throws hundreds of randomised, structured inputs at the function), no test for edge cases (non-breaking space, currency symbol, trailing whitespace, comma decimal separator), no integration test against actual scraper output, and no end-to-end test that would have caught "milk costs nothing" as a sanity violation in production-like data.

(b) Test plan for the price-cleaning function (using the Arrange-Act-Assert pattern):

  1. Happy path — "€1.50" → 1.50; "1.50€" → 1.50; "1,50 €" (comma as decimal, Spanish convention) → 1.50.
  2. Edge cases — "€0.99" → 0.99; "€1,234.56" (thousands separator) → 1234.56; "€ 1.50 " (extra whitespace) → 1.50; "€1.50\u00A0" (trailing non-breaking space) → 1.50.
  3. Adversarial — "" → throws or returns null with a documented contract; "abc" → same; "€" → same; "€1.5.0" → same.
  4. Property test — for any positive number n with at most two decimals, clean(format(n)) returns n. Run a thousand iterations.
  5. Sanity guard — an integration check that runs after each nightly scrape: if the median price of the milk category drops more than 30% night-over-night, halt publication and page on-call. This is not really a unit test, it's a production canary, and it's the only thing that would have caught the bug at the right time.

(c) The pyramid (Module E.7) is the right shape because unit tests are cheap and fast — the cleaning function should have hundreds of cases. But Module E also stresses that unit tests alone don't catch the real-world ones: scraper returns nationally-formatted prices, function written by someone who learned regex on English text, edge case lurks for years until a refactor brushes it. Integration tests against fixtures from the actual scrapers, plus a sanity guard at the data-pipeline level, complete the safety net.

Choosing the stack for a delivery start-up

Modules touched: D (architectures, mobile)

Scenario. A two-founder Madrid start-up wants to launch a same-day delivery app for paddle-club booking. They have nine months of runway, two engineers (one strong on JavaScript, one strong on the Java virtual machine), and a hypothesis they need to test fast. They expect 80% of their early users on Android, 20% on iPhone Operating System. The app needs: a sign-up flow, a paddle-court calendar, push notifications, and payments. They have no need for advanced graphics processing unit features and no plan for offline-first.

A friendly investor recommends they "go native, you'll need it later". Another friend recommends Flutter. The technical co-founder is leaning toward React Native because of the JavaScript skill in the team.

Tasks. (a) Analyse the trade-offs against the constraints. (b) Recommend a path, with a rationale. (c) Describe what evidence would later justify changing the path.

Model answer.

(a) "Native first" doubles the team's surface area: two codebases, two release pipelines, two skill stacks. With one strong JavaScript engineer, that's months of ramp-up on Swift or Kotlin. Flutter is excellent and would give a single codebase and great performance — but the team has zero Dart experience, and the runway is nine months. React Native uses skills the team already has, ships to both stores from one codebase, and has mature integrations for Stripe, push notifications via Firebase Cloud Messaging, and calendar widgets. The trade-off: React Native's performance ceiling is lower than Flutter's, and complex animations or graphics-processing-unit-bound features are awkward — but the brief excludes those.

(b) Recommend React Native, with the discipline that platform-specific code (the few cases where iPhone Operating System and Android genuinely differ — payments, push registration, deep linking) be isolated behind a small adapter layer so each platform keeps its idioms. Ship Android first (80% of the early audience) and iPhone Operating System a few weeks later. Use Expo if and only if the team is sure they won't need a native module Expo doesn't yet wrap; otherwise plain React Native. Distribute via beta channels — TestFlight for iPhone Operating System, Google Play internal testing — to a small pilot before the public launch.

(c) Evidence that would justify changing path: (i) consistent users complaints about input lag in the calendar grid that profiling traces to the JavaScript bridge, not to a fixable bug — at that point, write the calendar in native and embed it; (ii) animation requirements arrive that React Native's renderer can't hit at 60 frames per second on mid-range Android — Flutter rewrite worth considering; (iii) the team grows and one of the new hires is a strong native engineer — splitting platforms becomes affordable; (iv) the product becomes graphics-processing-unit-heavy (live video, augmented reality) — native is then non-negotiable. Module D's framing applies: pick the lightest architecture that fits the current problem, and let evidence drive the next move.

Polyglot persistence at a music-streaming company

Modules touched: C (databases), D (architectures)

Scenario. A Madrid-based music-streaming company is rebuilding its catalogue platform. The product team's requirements: (i) artist/album/track metadata that's heavily relational (an album belongs to an artist, a track belongs to an album, a track has many genres); (ii) full-text search across artist, album, track, lyrics; (iii) "people who liked this also liked" recommendations; (iv) playback events at roughly 2,000 per second across the user base, never lost, used both in real-time (now-playing widgets) and in batch (royalty calculations); (v) personalised home-page tiles that must render in under 100 milliseconds.

The architect proposes: PostgreSQL for the catalogue, Elasticsearch for search, Neo4j for recommendations, Apache Kafka for the event stream, Apache Cassandra for the event archive, and Redis for the home-page cache. The chief financial officer asks why this can't all live in PostgreSQL.

Tasks. (a) Justify each store choice on its requirement. (b) Address the chief financial officer's objection honestly — what's the cost of polyglot persistence? (c) Identify what should be a single source of truth across the system, and how it stays consistent.

Model answer.

(a) PostgreSQL — the catalogue is genuinely relational, has integrity constraints (a track can't exist without an album), and benefits from Atomicity-Consistency-Isolation-Durability transactions (e.g. when reassigning a track between albums). Elasticsearch — full-text search with relevance ranking, fuzzy matching, faceting, and language-aware tokenisation isn't something a relational database does well at scale; Elasticsearch is purpose-built. Neo4j — collaborative filtering and "shortest path" reasoning across an artist-listener graph express naturally as graph traversals; doing this in Structured Query Language requires recursive Common Table Expressions that are slow on large graphs. Apache Kafka — Kafka is a durable, ordered log designed precisely for event streams of this scale, with consumer groups so multiple downstream systems (real-time and batch) read independently. Apache Cassandra — write-heavy, time-series-shaped data with predictable access patterns (events for user X over time window Y) suits Cassandra's wide-row model; PostgreSQL would buckle on writes. Redis — sub-millisecond reads keyed on user identifier for the home-page tiles; PostgreSQL with proper indexing can do 5–10 milliseconds, Redis routinely does well under one millisecond.

(b) The chief financial officer is right that polyglot persistence has costs: more operational surface (six systems to monitor, patch, tune), more failure modes (cross-store consistency is now an application concern), more skills to hire for, more migration paths when something needs to change. The honest answer is that for a smaller product you absolutely should consolidate — PostgreSQL with the full-text search extension, plus Redis for cache, gets you remarkably far. The case for polyglot is evidence-based: you do it when a single store is demonstrably inadequate for a real, current requirement, not because it's fashionable.

(c) The catalogue (PostgreSQL) is the single source of truth for metadata. Elasticsearch and Neo4j are derived projections, fed from change-data-capture or scheduled synchronisation; if they get out of step, you reindex from PostgreSQL. The event stream (Kafka) is the single source of truth for "what happened"; Cassandra is its long-term archive, Redis is its short-term cache. Treating each store as either truth or projection is the discipline that keeps polyglot from becoming chaos.

The gym chain with the leaked password file

Modules touched: E (security), C (databases)

Scenario. A Madrid gym chain — fifteen branches, around 40,000 active members — discovers that an old back-up of its members table has been listed for sale on a forum. The table has columns including email, password_hash, phone, address, dni (Spanish national identifier). The "password hash" turns out to be unsalted MD5 (Message Digest 5).

Tasks. (a) Explain why unsalted MD5 is, in 2026, equivalent to plaintext. (b) Walk through the immediate response from the gym chain's perspective. (c) Walk through the response advice you'd publish for affected members. (d) Recommend the password-storage scheme they should adopt going forward.

Model answer.

(a) MD5 was designed in 1991 as a fast hash function, and "fast" is exactly the wrong property for password storage. Commodity graphics-processing units now compute billions of MD5 hashes per second. Unsalted means all users with the same password share the same hash, and "rainbow tables" — pre-computed lookups from common hashes back to plaintext passwords — turn the file into a dictionary attack at near-zero cost. For passwords that appear in any of the major breach lists (Have I Been Pwned catalogues these), the lookup is instant. For "12345678", "password", "real_madrid", "barcelona93" and so on, the attacker doesn't need to compute anything.

(b) Gym chain response, in order: (1) verify the breach is real by sampling and confirming hashes match the table; (2) force a password reset on every account, with sessions invalidated, before announcing publicly so attackers can't race the reset; (3) notify the Agencia Española de Protección de Datos within 72 hours under the General Data Protection Regulation, and notify members directly because the risk is high; (4) preserve forensic evidence and engage law enforcement; (5) engage a credit-monitoring or identity-monitoring partner if Spanish national identifiers were exposed, because those are reusable in fraud for years; (6) post-mortem and remediation as below.

(c) Advice for members: change the gym password immediately (it's already gone, but the reset closes the account); change that same password anywhere else you used it — credential stuffing across services is the larger danger; turn on multi-factor authentication (also called two-factor authentication) on email and bank, where a leaked password is most damaging; consider a password manager so re-using passwords becomes structurally impossible.

(d) Going forward: hash with bcrypt or Argon2 (Argon2id is the current default the password-hashing competition recommends), with a per-user random salt, and a work factor calibrated to take roughly 250 milliseconds on the production hardware. Never store the password in any form; never log it; and check new passwords against the Have I Been Pwned password list so users can't choose one already known to be compromised.

Test plan for a sign-up endpoint

Modules touched: B (Express), E (testing)

Scenario. A bike-share company (think BiciMAD) is adding a new sign-up endpoint:

POST /api/users
Body: { email, password, displayName, dateOfBirth, acceptedTermsVersion }
Constraints:
- email: valid syntax, not already registered
- password: ≥10 chars, not in the breach list
- displayName: 2–40 chars, no profanity from the configured list
- dateOfBirth: must be 16+ years old today
- acceptedTermsVersion: must equal the current published version
On success: 201, returns { userId }
On any failure: 400 with a structured error body

Task. Produce a test plan covering all three layers of the pyramid (Module E.7).

Model answer.

Unit tests (run in milliseconds, hundreds of cases, no input/output):

  • Password validator: too short, exactly 10, exactly 11, contains spaces, contains non-Latin script (must accept), known-breached value (must reject), 1,000-char password (must accept or reject by documented contract).
  • Display-name validator: too short, too long, exactly at boundaries, contains profanity from the list, contains profanity in unicode look-alike characters (depending on policy), trims whitespace, rejects empty after trim.
  • Age validator: exactly 16 today, one day under, one day over, leap-year birthday, dates in the future, malformed date strings.
  • Terms-version validator: equal, off-by-one, null.

Integration tests (run in seconds, dozens of cases, real Express app + test database):

  • Happy path: valid body → 201 with a user identifier; row exists in database with hashed password (never plaintext).
  • Email-uniqueness: insert a user, attempt re-registration → 400 with the right error code, no row created (Atomicity).
  • Breached password: known-breached input → 400, no row created.
  • Concurrent-registration race: two simultaneous requests with the same email → exactly one succeeds.
  • Database failure: simulate database unreachable → 500 with no leaked stack trace, request logged.
  • Rate limit: 30 requests in 60 seconds from one internet protocol address → some return 429.

End-to-end tests (run in tens of seconds, a handful of cases, real browser via Playwright or Cypress):

  • Open the sign-up page, fill the form with valid data, submit, verify redirect to a "check your email" page, verify the verification email arrives in the test inbox.
  • Submit with a too-young birthdate, verify the inline validation message appears in the right language.
  • Submit, wait for the welcome email, click the verification link, verify the account is now activated and the user can sign in.

Out-of-pyramid checks:

  • A security-focused test that submits 1,000 nonsense bodies and asserts no 500 errors leak.
  • A property-based test on the password validator that holds for any string of length 10–100.
  • A canary in production that signs up a synthetic user every 10 minutes and pages on-call if it fails.

The pattern in all of these is Arrange-Act-Assert: set up the world, do the thing, check the consequence.

Migrating a newspaper from multi-page to single-page

Modules touched: D (architectures)

Scenario. A regional Spanish newspaper runs a 12-year-old PHP multi-page application. It's slow on mobile (each article reload is 1.8 seconds median), but it ranks well in search (because every article is a real, server-rendered hypertext markup language page with proper meta-tags) and the editorial team writes in a content-management system that publishes static-ish hypertext markup language. A new chief technology officer proposes a full rewrite as a Vue single-page application. The editorial leadership is nervous: "we are not a mobile app, we are a newspaper, search rankings are our oxygen."

Tasks. (a) Explain why the editorial leadership's nervousness is well-founded. (b) Recommend an architecture that wins both fights — speed and search-friendliness. (c) Sketch a migration plan that doesn't risk the whole site at once.

Model answer.

(a) A pure single-page application renders an empty hypertext markup language shell on first load and fills it in with JavaScript. Search-engine crawlers — including Google's — handle JavaScript better than they used to, but not flawlessly: render budget is finite, slow JavaScript means the crawler may move on before content appears, and link-graph signals can degrade. For a newspaper whose business depends on appearing high in search results, "we'll be fine, Google handles JavaScript now" is a real risk to pin the company on.

(b) Use server-side rendering with a framework like Nuxt (the Vue-flavoured version of Next.js) or, for an even more static approach, a JAMstack generator like Astro that pre-renders article pages as static hypertext markup language at publish time. Either way, the first paint is real hypertext markup language with the article text inside it — search engines see an article, mobile readers see content in under a second — and JavaScript progressively enhances the experience with comments, infinite scroll, and personalisation. This gets the user-experience benefits the chief technology officer wants without sacrificing the editorial team's oxygen.

(c) Migration plan: (1) carve out one section first — say, the football blog — and rewrite it on the new stack, leaving the rest of the site on PHP behind a path-based reverse-proxy split; (2) measure: page-load times, search rankings for that section's keywords over four weeks, error rates; (3) only if the metrics hold, migrate a second section, then a third; (4) the politically loaded section (national politics, with the most search volume) is migrated last, when the operational pattern is well understood; (5) keep the content-management system stable through the migration — editors should not feel any disruption — by having both stacks read from the same source. The key is that no migration is a single rewrite; every migration is a sequence of small, reversible moves.

A progressive web app for a thin-network market

Modules touched: D (architectures, mobile), B (web app fundamentals)

Scenario. A social-network company has launched in Western Europe and now wants to expand into a market where: (i) the median user is on Android with 1.5 GB random-access memory and an entry-level chip; (ii) data plans are 1 GB per month and the network is unstable on transit; (iii) the Google Play Store is the dominant install channel but installing yet another app is a friction the user actively resists. The product team is pushing for a native rewrite tuned for low-end hardware. An engineer proposes a progressive web application instead.

Tasks. (a) Make the case for the progressive web application choice against the constraints. (b) Identify the trade-offs the team must accept. (c) Specify the technical building blocks required.

Model answer.

(a) The constraints favour the progressive web application strongly. Install friction is low — adding a website to the home screen is one tap, no 80-megabyte download. Data usage is low — a well-built progressive web application is in the 200-kilobyte range on first load and far less on repeat visits because the service worker caches assets. The unstable network is exactly what service workers solve: the application keeps working when the connection drops, queues writes, and retries when the network comes back. On entry-level random-access memory, a progressive web application is lighter than a native app because it shares the browser process. This pattern was real and well-documented for Twitter Lite, which expanded to thin-network markets with a progressive web application instead of a native app.

(b) Trade-offs to accept: no access to advanced sensor application programming interfaces, no background tasks beyond what service workers permit, push notifications are weaker than native especially on iPhone Operating System, app-store discovery is partly lost (though installable progressive web applications are now indexed by Google Play and similar). For this product — a social network, no advanced sensor needs, no graphics-processing-unit-heavy features — none of these are dealbreakers.

(c) Technical building blocks: a web app manifest so the browser knows the app's identity, icon, and start URL; a service worker registered on first load, with a cache-first strategy for static assets and a network-with-cache-fallback strategy for application-programming-interface responses; a thoughtful image strategy (small, WebP, responsive); code-split bundles so first paint is small; offline write queueing using IndexedDB plus the Background Sync application programming interface; performance budget enforced in continuous integration so a bloated bundle fails the build. All of this is hypertext-markup-language plus cascading-style-sheets plus JavaScript on top of a server-rendering layer (Module D), one codebase, deployed as a website that happens to install.

Model answer

A web server is a long-running process that listens on a Transmission Control Protocol port — port 80 for plain Hypertext Transfer Protocol, port 443 for Hypertext Transfer Protocol Secure — and answers Hypertext Transfer Protocol requests with Hypertext Transfer Protocol responses. On every request the server parses the request line and headers, decides what the request is asking for (a static file, a dynamic computation, an application-programming-interface call), produces or fetches the response body, sets the appropriate status code and headers, and writes the bytes back over the socket.

The reason one Node.js process can serve thousands of users without one thread per user is the event loop plus non-blocking input/output. While the operating system fetches a file from disk or waits on a database reply, Node doesn't sit idle on a thread — it keeps accepting and progressing other requests, then resumes the original when the input/output completes. This works beautifully for the "sit and wait" workloads that dominate the web (database queries, network calls, file reads). It works less well for central-processing-unit-bound work, where a single slow handler can starve every other request, which is exactly the failure pattern Module A warns about.

Model answer

Apache HTTP Server is the older of the two, designed for modular configuration, with a long history of .htaccess overrides per directory and a vast ecosystem of modules. It traditionally used a process-or-thread-per-connection model, which is straightforward but limits raw concurrency. Apache shines on shared-hosting platforms and where per-directory rules genuinely help.

Nginx was designed event-driven from the start to solve the "C10K problem" — handling tens of thousands of simultaneous connections on modest hardware. It's typically faster at static-file serving and excels as a reverse proxy and load balancer in front of application servers like Node.js or Python's Web Server Gateway Interface (WSGI). For a modern stack — Node application behind a reverse proxy, terminating Transport Layer Security, cacheing static assets, distributing across multiple application servers — Nginx is usually the natural choice. For legacy applications already wired into Apache modules, or for simple shared hosting, Apache remains entirely reasonable.

Model answer

In Express, routing is the dispatch from (method, path) to a handler — app.get('/users/:id', getUser). The handler reads req.params.id, fetches the user, calls res.json(user). Middleware is any function with the signature (req, res, next) that runs as part of the request/response pipeline. It can mutate the request, write to the response, log, authenticate, or short-circuit by calling res.send() instead of next().

Middleware is central because nearly every cross-cutting concern in a web application is a middleware: body parsing (express.json()), session handling, authentication, request logging, rate limiting, error handling. Express composes middleware as a chain — app.use(...) queues a function; the first to call res.send() ends the chain; errors propagate to the error-handling middleware (the four-argument form). Once you internalise that middleware is just a function that runs in a chain, you stop thinking of it as magic and start writing your own. That mental model is the difference between using Express and understanding Express.

Model answer

Node.js provides the JavaScript runtime — V8 plus the event loop plus standard libraries for files, sockets, and Hypertext Transfer Protocol. Express provides a thin layer on top that turns "raw Hypertext Transfer Protocol parsing" into "declare a route and a handler". npm provides the ecosystem: instead of writing body parsing, password hashing, JavaScript Object Notation Web Token validation, application-programming-interface clients, and database drivers from scratch, you npm install battle-tested libraries.

The weekend-build pattern is something like: scaffold a project with npm init; install Express, a database driver, an Object-Relational Mapper or query builder, a validation library, bcrypt, helmet for security headers, morgan for request logging; wire them together as middleware; write a few routes for the core resources; deploy to a managed platform like Render, Railway, or Fly.io. The flip side, which Module E hammers home, is that "wire together npm packages" without a security mindset is exactly how you ship the OWASP Top Ten by accident.

Model answer

Vue's reactivity is a contract: declare your data as reactive, and any template that reads that data will re-render automatically when the data changes. You don't write document.getElementById('counter').innerText = newValue — you write count.value++, and Vue figures out which parts of the user interface depend on count and updates exactly those.

What this gives the developer is a one-way mental model: the user interface is a function of state. To change the user interface, change the state. To debug, look at the state. Vanilla document-object-model manipulation, by contrast, is a sea of little update calls scattered across event handlers — easy to forget one, easy to set the same element from two places that disagree. Reactivity also enables computed properties (values derived automatically from other reactive values) and watchers (side-effects triggered when a reactive value changes), both of which would be tedious to implement by hand. The cost is a small amount of magic; the benefit is whole categories of bugs that simply cease to exist.

Model answer

Atomicity-Consistency-Isolation-Durability gives you a contract: your transaction either fully happens or doesn't (Atomicity), the database moves between valid states (Consistency), concurrent transactions don't see each other's intermediate state (Isolation), and once committed your write survives a power cut (Durability). For systems where correctness is the headline property — banking, ticket sales, medical records, anything where money or identity changes hands — ACID is the table stakes.

You give up ACID — partially or wholly — when the workload's nature makes the cost of those guarantees too high relative to the value. A click-tracking system that ingests two million events per second can tolerate a lost event here or there; demanding strict immediate consistency across a globally distributed cluster would either cost a fortune or fail to meet latency targets. Eventual consistency (the BASE model in NoSQL stores like Cassandra or DynamoDB under specific configurations) is a deliberate trade: availability and partition-tolerance during network splits, at the cost of "the value you read might be a few hundred milliseconds stale". Knowing which trade your system can afford is the central database design decision Module C trains you to make.

Model answer

The Consistency-Availability-Partition-tolerance theorem (Eric Brewer, 2000) observes that in a distributed system, when the network partitions — and it will, eventually — you cannot have both strict immediate consistency and full availability. Pick two; partition-tolerance is non-negotiable in any system spread across machines, so the real choice during a partition is C versus A.

Basically Available, Soft state, Eventual consistency reframes the database contract for distributed systems that prefer A over strict C. "Basically available" means the system answers requests under partition; "soft state" acknowledges that the values you observe may be stale or may converge; "eventually consistent" promises that, given enough time without new writes, all replicas will agree.

This reframes our expectations: ACID is the gold standard for single-machine and tightly-coupled systems; BASE is the realist's contract for systems that span data centres or continents. When you pick MongoDB, Cassandra, or DynamoDB, you're picking the BASE side of the trade — and you have to design your application logic to be tolerant of that, with idempotent writes, last-writer-wins or vector-clock conflict resolution, and an honest acceptance that "the database is the source of truth, exactly" is a simplification a globally distributed product can't always afford.

Model answer

Classical relational thinking assumes a known schema, fits-in-one-machine data, and queries that benefit from indexes the database administrator has chosen well. Big Data thinking starts from the assumption that one or more of those is broken: the data doesn't fit on one machine, the schema isn't known up front, the data arrives faster than a single process can write it, the queries are exploratory, and the analysis often runs on a different cadence than the operational workload.

The patterns Big Data brings — distributed file systems like the Hadoop Distributed File System, parallel processing frameworks like Apache Spark, streaming platforms like Apache Kafka, columnar warehouses like Snowflake, BigQuery, Redshift, and the lakehouse vision from Databricks — all share the move from "single machine, structured, transactional" to "fleet of machines, semi-structured, analytical". The five Vs (Volume, Velocity, Variety, Veracity, Value) name what's hard. The MapReduce family of execution models name how you get work done across a fleet. None of this replaces the relational store — your operational system still wants Atomicity-Consistency-Isolation-Durability — but it complements it for analytics, machine learning, and exploratory work that the relational store was never designed for.

Model answer

For an early-stage start-up, a well-modularised monolith is almost always the right call. One codebase, one deployment pipeline, one database, one log to read when something breaks. Refactoring is a single repository's worth of work, not a distributed-systems transaction. The team is small enough that "deploy independently" isn't yet a benefit anyone can claim.

Microservices solve organisational problems that early-stage start-ups don't have: many teams stepping on each other in one repo, services with wildly different scaling profiles, language or runtime requirements that diverge by domain, organisational accountability per service. Their costs are real: distributed tracing, service discovery, network as a failure mode, deployment of multiple independent units, schema evolution across services, harder local development. The honest framing in Module D is that microservices are a response to scale and team structure, not a starting point. Start as a monolith, design module boundaries with future extraction in mind, and split a service out only when you can name the specific pain that splitting solves. Premature microservices have killed more start-ups than monoliths have.

Model answer

A Multi-Page Application (MPA) renders a fresh hypertext markup language page on every navigation. It's the original web pattern, it's what search engines understand best, and for content-heavy sites where each page is essentially a document — newspapers, blogs, government portals — it remains an excellent default. The downside is that each navigation costs a full round-trip and a re-render.

A Single-Page Application (SPA) loads a JavaScript shell once and updates the user interface in place as the user navigates. It feels app-like — instant transitions, persistent client-side state — and is well-suited to logged-in dashboards, editors, and software-as-a-service products where the same user spends a long session inside the application. The downsides are slower first paint, weaker search-engine experience, and the operational complexity of state management.

Server-Side Rendering (SSR) — usually via a framework like Next.js for React or Nuxt for Vue — gives the best of both: real hypertext markup language on first load (so the page is fast and indexable), then the JavaScript hydrates and behaves like a SPA for subsequent navigation. It's the right answer for sites that are simultaneously content-heavy and app-like — e-commerce being the canonical example, where the product page must rank in search and the cart must feel interactive. The cost is more complexity in the build and deploy pipeline.

The decision tree, simplified: content site, search matters, sessions short — MPA or static. Logged-in app, sessions long, search not part of the value proposition — SPA. Content matters and interactivity matters — SSR. JAMstack is the special case where the content is mostly static and you can pre-render at build time.

Model answer

Start with the question that decides everything: does the product genuinely need platform-specific capabilities (advanced sensors, the augmented-reality kit, deep operating-system integrations, premium graphics-processing-unit performance)? If yes, native — Swift on iPhone Operating System, Kotlin on Android — is the answer, with the cost of two codebases.

If not, the next question is reach versus richness. If the product is essentially a website that benefits from being on the home screen — news, social, shopping — a Progressive Web Application (PWA) is the lightest path: one codebase, no app store gatekeeper, installable, offline-capable. The famous case is Twitter Lite for thin-network markets.

If the product needs an app-store presence and reasonably native feel but the team is small and writes JavaScript, cross-platform (React Native, Flutter) gives one codebase with near-native feel. Hybrid (Cordova, Ionic) wraps a web view — cheaper still, but the user-experience ceiling is lower; useful for internal apps or quick prototypes.

A pure web app (no install) is the right choice when "open this in a browser" is acceptable to your users — internal tools, low-friction software-as-a-service, anything where install friction is a non-starter.

The error to avoid is "we should go native because real apps are native". Real apps are whatever fits the user, the team, and the runway.

The Top Ten 2021 lists Broken Access Control (A01), Cryptographic Failures (A02), Injection (A03), Insecure Design (A04), Security Misconfiguration (A05), Vulnerable and Outdated Components (A06), Identification and Authentication Failures (A07), Software and Data Integrity Failures (A08), Security Logging and Monitoring Failures (A09), and Server-Side Request Forgery (A10).

The three most under-appreciated, in my reading: A04 Insecure Design, because junior developers think of security as a list of bugs to fix, when the deepest vulnerabilities are architectural — the wrong trust boundary, the wrong default, the wrong assumption about who calls whom; A05 Security Misconfiguration, because it doesn't feel like a vulnerability — the code is fine — but a database left listening on a public interface or a cloud-storage bucket left world-readable is exactly how the headline breaches happen; and A09 Security Logging and Monitoring Failures, because it's the dog that doesn't bark — without it, attackers operate inside your system for months, and dwell time is the metric that decides whether an incident is a near-miss or a disaster.

The framing that makes Module E click is that the Top Ten isn't ten bugs — it's ten categories of mistakes the industry keeps making, and the antidote is design discipline plus defence-in-depth, not a checklist at the end of the project.

Model answer

Structured Query Language injection happens when user input is treated as Structured Query Language code rather than data. The classic case is concatenating a username into a query: "SELECT * FROM users WHERE name = '" + username + "'". If username is ' OR '1'='1, the query becomes SELECT * FROM users WHERE name = '' OR '1'='1', returning every user. Worse payloads can drop tables, exfiltrate data, or escalate privileges via the database's own features.

The naive defences — strip dangerous characters, escape quotes, allow-list known-good inputs — all fail on the same systemic ground: you're trying to parse a language and remove its dangerous parts before passing the result through another parser, the database's. Every escape function is a partial implementation of the database's parser, and partial parsers always have edge cases. Different databases have different escape rules. New attack patterns are found regularly. The escape-it-yourself path is a treadmill.

Parameterised queries (prepared statements) sidestep the whole game. The query template — SELECT * FROM users WHERE name = ? — is sent to the database as Structured Query Language. The user input is sent as a separate, typed parameter. The database never parses the user input as Structured Query Language, full stop. There's no injection because there's no concatenation. Object-Relational Mappers like Sequelize, Prisma, and TypeORM use parameterisation under the hood, so "use the ORM properly" is a derivative of the same rule. The only sound defence is structural separation of code and data; everything else is patching the symptom.

Model answer

The three flavours: Reflected XSS — the payload is in the request (typically a query-string parameter) and the server echoes it back into the response without escaping; the attack vector is a crafted link the victim clicks. Stored XSS — the payload is saved server-side (in a database, a comment, a profile field) and served to subsequent visitors; the attack vector is anyone who views the affected page. DOM-based XSS — the payload is processed entirely in the browser, by JavaScript that takes user-controlled input (location.hash, for instance) and feeds it into a sink that creates hypertext markup language (innerHTML); the server may never see the malicious input.

The layered defences: (1) output encoding — every place that puts user data into hypertext markup language, JavaScript, attributes, URLs, or cascading style sheets must encode it for that context. Modern templating engines and frameworks (Vue's {{ }}, React's JSX text children) do this by default; the danger is the explicit-escape-hatch (v-html, dangerouslySetInnerHTML) and the document-object-model sink. (2) Content-Security-Policy header — a strict CSP that disallows inline scripts, unsafe-eval, and unexpected origins refuses to execute injected code even if encoding fails. (3) HttpOnly and SameSite cookies — limit the damage of XSS by stopping JavaScript from reading the session cookie and limiting cross-site cookie attachment. (4) input validation — useful as defence-in-depth but never sufficient on its own; a comment legitimately containing < is not a security vulnerability, an unencoded comment containing < is.

The mental model from Module E: XSS is a trust problem (we trusted user content) and a context problem (we put it in the wrong context unescaped). The fixes are correspondingly two layers — don't trust, encode for context.

Model answer

End-to-end correct password handling means: passwords are sent only over Hypertext Transfer Protocol Secure; the server never logs them; the server never stores them in any reversible form; the server hashes them with Argon2id (or bcrypt at a current cost factor) using a per-user cryptographically-random salt; the work factor is calibrated to roughly 250 milliseconds on production hardware and is increased over time as hardware gets faster; new passwords are checked against Have I Been Pwned's password list and rejected if known-breached; minimum length is at least 10 characters with no upper limit and no perverse complexity rules that push users toward "Pa$$word1!"; multi-factor authentication is offered (and required for sensitive accounts) using time-based one-time-password applications or hardware security keys (WebAuthn / passkeys are the modern path); password reset flows use single-use, time-limited, unguessable tokens; failed-login attempts are rate-limited per account and per internet protocol address, not silently letting attackers brute-force; sessions issued on login are bound to a cookie with Secure, HttpOnly, and SameSite=Lax flags, regenerated on privilege change, and explicitly invalidated on logout.

The reason every one of these matters individually is that any single failure — plaintext storage, no salt, fast hash, breached password accepted, no rate limit, leaked session cookie — is enough to compromise users. The reason they reinforce each other is defence-in-depth: even if one layer is buggy, the next catches the attack. The phrase to remember is "users will choose weak passwords; design for that reality, don't fight it."

Model answer

Hypertext Transfer Protocol Secure (HTTPS) wraps Hypertext Transfer Protocol in Transport Layer Security, providing three guarantees on the wire: confidentiality (an eavesdropper can't read the payload), integrity (a tamperer can't silently modify it), and authentication (the client verifies the server's certificate against a trusted certificate authority). The certificate-authority ecosystem — modernised by Let's Encrypt into a free, automated default — means the cost objection to "always Hypertext Transfer Protocol Secure" no longer exists.

Hypertext Transfer Protocol Strict Transport Security (HSTS) is a response header that tells the browser, for a configured duration, to refuse to connect over plain Hypertext Transfer Protocol — even if the user types http://, even if a man-in-the-middle tries to downgrade. It blocks the "first connection over plain Hypertext Transfer Protocol" attack window after the first secure visit, and the HSTS preload list eliminates even the first visit by baking the rule into browser binaries.

Content-Security-Policy (CSP) is a different layer: it controls what the browser is allowed to execute and load once content arrives. A strict CSP says "scripts only from this origin, no inline scripts, no eval", which defangs most Cross-Site Scripting payloads even if your encoding has a hole.

Together they form a story: HSTS makes sure you got the right page; HTTPS makes sure the page wasn't tampered with in transit; CSP makes sure that even if the page itself contains injected hostile content, the browser refuses to run it. Each layer covers a different failure of the others. Module E's broader argument is exactly this: security isn't a checkbox, it's a stack of layers each compensating for the others' gaps.

Model answer

The pyramid prescribes many fast unit tests at the base, fewer slower integration tests in the middle, and a small number of end-to-end tests at the top. The shape encodes two facts. First, cost grows as you climb: a unit test runs in milliseconds with no input/output and no setup; an integration test needs a database and a real server; an end-to-end test launches a browser, navigates, waits for animations. Second, value per test — meaning, "if this test fails, how clearly does it tell me where the bug is?" — also changes: a failing unit test points at a function; a failing end-to-end test points at the entire system, and you spend an afternoon figuring out which layer broke.

When teams invert the pyramid — many slow end-to-end tests, few unit tests — three things go wrong. The continuous-integration pipeline becomes slow, so engineers stop running tests locally and the feedback loop collapses. End-to-end tests become flaky (browsers, networks, timing) and the team starts ignoring failures or rerunning until green, which destroys the signal. Bug localisation becomes painful, so debugging eats the time that should have gone into building. The pyramid's logic is that each layer should catch the bugs it's best at catching: pure logic at the unit level, contract integration at the integration level, critical user journeys at the end-to-end level.

The honest extension Module E adds is that the pyramid is a guideline, not a rigid prescription. Some products (anything with a heavy front-end) need more end-to-end coverage proportionally. Some pure-back-end services need almost none. The discipline is to know why you're putting a test at the layer you're putting it at, not to defer to the diagram.

Model answer

Arrange-Act-Assert (AAA) is a three-section structure for a unit test: arrange the world the test needs, act on the system under test, assert the consequence. It's a readability convention more than a technical one — separating these phases makes a test easy to read, modify, and reason about.

Worked example. Suppose calculateOrderTotal(items, discountCode, country) returns the total in euros, applies a percentage discount if the code is valid and not expired, and applies value-added tax based on country.

test('applies a 10% discount and Spanish value-added tax of 21%', () => {
  // ARRANGE
  const items = [
    { sku: 'A', price: 50, qty: 2 },   // 100
    { sku: 'B', price: 25, qty: 1 },   // 25
  ];
  const discountCode = 'SPRING10';     // valid, not expired
  const country = 'ES';

  // ACT
  const total = calculateOrderTotal(items, discountCode, country);

  // ASSERT
  // subtotal 125, less 10% = 112.50, plus 21% VAT = 136.13 (rounded to cents)
  expect(total).toBeCloseTo(136.13, 2);
});

The arrange section makes the inputs explicit; the act section is one line so any failure points clearly to the boundary; the assert section says, in words and numbers, what the expected outcome is and why. Each test should ideally have one logical assertion — multiple expect calls are fine if they all describe the same conceptual outcome, but a test that checks five unrelated things is really five tests in disguise. The pattern's real power is that, six months later, you can read the test and immediately understand both the setup and the contract being verified.

Model answer

Mobile testing inherits everything web testing covers (unit, integration, end-to-end, accessibility, performance) and adds at least four concerns the web rarely faces. Device fragmentation — Android in particular spans thousands of device models, screen sizes, manufacturer skins, and operating-system versions still in active use; you can't test on every one, so you build a device matrix representative of your audience (top market shares + outliers) and run on real devices via Firebase Test Lab or similar cloud farms. Network conditions — phones move, signals weaken, the radio drops. Tests must simulate edge-of-cell, transition between Wi-Fi and cellular, complete loss of connectivity, and the behaviour expected (queue writes, show offline state, retry on reconnection). Network Link Conditioner on iPhone Operating System and the Android emulator's network controls are the standard tools.

Gestures and interactions — taps, swipes, long-presses, pinch-zoom, multi-touch. Web tests rarely cover these well; mobile frameworks like Appium, Detox, and Maestro treat gestures as first-class. Battery and central-processing-unit profiling matter too — an app that drains the battery faster than competitors will lose users regardless of how correct its features are. Permissions and platform integrations — camera, location, notifications, contacts each require explicit user consent at a moment determined by the operating system, and your tests have to simulate "user grants", "user denies", "user grants then revokes later".

The pyramid still applies — unit tests at the base, end-to-end at the top — but the top is more expensive than on the web, so investing more in unit and integration coverage of the platform-specific code (deep-link handling, push registration, in-app purchase flows) pays back.

Model answer

Three months is enough for focus but not for everything; the priorities have to be ruthless.

Month 1 — Stop the bleeding. Audit the application against the OWASP Top Ten: are there Structured Query Language Injection vectors (Module E.3)? Cross-Site Scripting risks (E.4)? Plaintext or weak password storage (E.5)? Plain-Hypertext-Transfer-Protocol exposure (E.6)? Fix these in priority order — they're the categories that cause headlines. Put the application behind a reverse proxy (Module A) with rate limiting and Hypertext Transfer Protocol Secure terminated by Let's Encrypt. Set basic security headers (helmet handles this). Add request logging with structured fields and a request identifier propagated end-to-end so the next incident can be traced.

Month 2 — Make it observable and testable. Write unit tests around the most-changed code paths (Module E.7, E.8) — not because every line needs coverage but because the next refactor will need a safety net. Add integration tests for the application-programming-interface contracts, especially payments and authentication. Add a continuous-integration pipeline that runs tests on every push (cheap, transformative). Add basic application-performance monitoring — even a free tier of Sentry for errors and Better Stack or Grafana Cloud for metrics. Look at the database (Module C): is the schema sane? Are there indexes on the columns the hot queries filter on? Are back-ups actually working and tested by restore?

Month 3 — Architectural debt, intentionally. With observability in place, you can see what hurts. Decide whether the architecture needs server-side rendering for marketing pages (Module D.2) — usually yes if search traffic matters. Decide whether the back-end should split a service out (Module D.1) — usually no, but document the seams for future. Decide whether the mobile presence should be a Progressive Web Application (Module D.3) — often a strong yes for small businesses, low cost, real value. Set a small set of error-budget objectives ("payment endpoint 99.9% successful over 30 days") so the team knows what "good" looks like.

The cross-cutting discipline is documented in Modules A through E, but lived rather than learned: assume hostility from inputs, design for failure, observe what runs, test what you ship, and keep the smallest architecture that fits the current problem. Three months done with focus beats twelve months done without it.
