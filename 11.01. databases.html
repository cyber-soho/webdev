The Foundations of Memory

This section introduces the core concept of databases and their humble beginnings. A database is simply "a place that remembers at scale." We explore the four fundamental reasons applications rely on dedicated database management systems rather than simple spreadsheets or files.

The 4 Reasons We Use Databases

💾

Persistence

Data survives power loss. It remains intact beyond the application's lifecycle.

👥

Concurrency

Many users can read and write at the exact same time without data collisions.

🛡️

Integrity

Strict rules are enforced. Invalid data or garbage is refused at the door.

⚡

Performance

Indexes allow finding a single record among billions almost instantly.

📄 Flat Files: The Honest Ancestor

CSV, TSV, and JSONL are plain text files, one record per line. They are the "Esperanto of data" - universally readable.

✅ When to use (The Tool)

  • Read-mostly configs (.env, .yaml)
  • Data interchange (nightly CSV dumps)
  • Fits in memory (tens of thousands rows)

❌ When to avoid (The Trap)

  • Concurrent writes (causes corruption)
  • Indexed lookups across millions of rows
  • Need transactions or referential integrity

SQL: The Fifty-Year-Old King

Structured Query Language is the undisputed industry standard. This section explores why declarative querying, strict relational models, and ACID transaction guarantees make SQL the default choice for the vast majority of software projects, from banking to e-commerce.

The Declarative Magic

You tell the engine what you want, not how to get it. Decades of algorithmic optimization happen automatically.

SELECT full_name
FROM Student
WHERE country = 'Spain';

You write 8 words. The engine handles the billions of rows.

The ACID Contract

Atomicity
All or nothing. No half-finished transactions.
Consistency
Data always moves from one valid state to another.
Isolation
Concurrent transactions act as if they are alone.
Durability
Once committed, it survives power loss.

NoSQL: Scale, Schema, Speed

When massive internet companies hit the physical limits of single servers around 2005, NoSQL ("Not Only SQL") emerged. This section details how trading strict consistency for horizontal scalability birthed four specialized database families designed for specific data shapes.

📄

Document Stores

Self-describing JSON-like records. Schema-flexible.

Use: CMS, Catalogs, Mobile apps
Ex: MongoDB, Firestore
🔑

Key-Value Stores

Simplest database. All operations O(1). In-memory speed.

Use: Caching, Sessions, Counters
Ex: Redis, DynamoDB
📊

Wide-Column Stores

Millions of sparse columns per row. Scales across datacenters.

Use: IoT, Logs, Time-series
Ex: Cassandra, Bigtable
🕸️

Graph Databases

Nodes and edges are first-class citizens. Fast multi-hop traversal.

Use: Social nets, Recommendations
Ex: Neo4j, Neptune

Hybrid Systems & Big Data

Modern applications rarely rely on a single database. This section looks at Polyglot Persistence (mixing tools like SQL, Redis, and Graph), and introduces the Big Data ecosystem designed to handle information that exceeds the capacity of traditional single-machine engines.

🧩 Polyglot Persistence

Using the best specialized tool for each distinct workload within a single application architecture (e.g., Spotify).

Users/Payments PostgreSQL
Sessions/Counters Redis
Music Search Elasticsearch
Recommendations Graph DB
