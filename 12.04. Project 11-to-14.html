Module 4 · Application Types & Architectures

11.

Web Application Architectures

≈ 1.5 pages in your report

Monolith MVC SPA Microservices Serverless PWA

Background and Why It Matters

The way a web application is structured — not just its code, but the very logic of how its components communicate and where processing happens — has profound consequences. It affects how fast the application loads, how easily a team can maintain it, how much it costs to scale, and how vulnerable it is to failure or attack.

Over the past two decades, the industry has moved from simple monolithic setups where a single server does everything, to distributed, event-driven, containerised ecosystems where dozens of independent services collaborate in real time. Understanding these patterns is not optional for a web developer — it is foundational.

1
Phase 1 · Theory & Context

Read actively. Sketch trade-offs.

≈ 45 min

Read actively. As you go through the concepts below, sketch a quick diagram for each pattern in your notebook or a digital tool of your choice. You do not need to be exhaustive — focus on understanding trade-offs.

1.1Monolithic Architecture

A monolithic application packages the user interface, the business logic, and the data access layer into one deployable unit. Early web applications — forums, e-commerce sites, content management systems — were almost universally built this way. The approach is straightforward to develop and deploy, but as the codebase grows, a single change requires redeploying the entire system.

Think about it from a pragmatic angle: for a solo developer or a small startup, a monolith is often the right choice. For a team of fifty engineers working on different features simultaneously, it becomes a bottleneck.

1.2MVC (Model-View-Controller) Pattern

MVC (Model-View-Controller) is a design pattern that separates an application into three interconnected components. The Model manages data and business rules. The View is what the user sees — the rendered output. The Controller receives user input, calls model logic, and selects the appropriate view. Most classical frameworks — Django (Python), Laravel (PHP), Ruby on Rails — implement some variant of MVC. Understanding it is essential before tackling anything more complex.

1.3SPA (Single-Page Application) Architecture

SPA (Single-Page Application) architectures load a single HTML shell and dynamically update the content via JavaScript without full page reloads. The browser takes over much of the rendering work that used to belong to the server. This dramatically improves perceived speed and interactivity. The trade-off is added complexity on the client side, potential SEO (Search Engine Optimisation) challenges, and an initial bundle size that needs careful management.

1.4Microservices Architecture

Rather than one large deployment unit, a microservices architecture decomposes an application into small, independently deployable services — each responsible for a single business capability. An e-commerce platform might have separate services for user accounts, the product catalogue, the shopping cart, payment processing, and email notifications. These services communicate over HTTP (Hypertext Transfer Protocol) REST (Representational State Transfer) APIs or message queues.

Benefits: independent scaling, technology diversity, fault isolation. Costs: network latency, operational complexity, the need for service discovery, and a steeper learning curve for distributed debugging.

1.5Serverless and Event-Driven Patterns

In a serverless model, the developer writes functions and deploys them to a provider that handles infrastructure, scaling, and availability entirely. Functions are invoked by events — an HTTP request, a file upload, a database change — and they run only for as long as needed. The developer pays for execution time rather than idle server capacity.

Research prompt: Look at how AWS Lambda, Google Cloud Functions, and Cloudflare Workers each implement this concept. You are not required to use any of them — the point is to understand the model and find the tool that fits your context.

1.6Progressive Web Applications (PWA)

A PWA (Progressive Web Application) bridges the gap between a web page and a native mobile application. It uses standard web technologies but adds capabilities — offline access via service workers, push notifications, installation to the home screen — that make the experience feel native. From an architecture perspective, a PWA typically combines a responsive SPA with a caching strategy managed by a service worker and a web manifest file.

Reference: web.dev/progressive-web-apps maintained by Google Chrome developers is a good starting point. As always, you are free to consult any equivalent resource you find reliable and up to date.

Phase 2 · Hands-on Lab

Build a minimal comparison prototype.

≈ 1 h 30 min

Objective: Build a minimal comparison prototype that demonstrates at least two different architectural approaches.

01
Choose your scenario.

Pick a simple application idea (examples: a to-do list, a weather dashboard, a basic blog). You will implement a skeleton of it in two architectural styles. Keep it small — you are not building a finished product; you are illustrating a concept.

02
Implement a monolithic version.

Create a server-side rendered version of your scenario. The server handles routing, fetches data (you can mock it with a local JSON file), and returns a complete HTML page. Any server technology is acceptable — Node.js with Express, Python with Flask, PHP, or anything else you find appropriate and can install on your machine.

Document every decision you make. Why did you choose this framework? What does the directory structure look like? A screenshot with annotations counts as documentation.

03
Implement an SPA version.

Convert your scenario so that it runs in the browser as a SPA. A very thin back-end (or even a static JSON file served locally) feeds data to a JavaScript front end. You may use any framework or none at all — Vanilla JS, Vue.js, React, Svelte, Angular, or any other tool you research and judge appropriate.

04
Draw a comparison diagram.

Produce a side-by-side architectural diagram of both versions. It does not need to be polished — a clear hand-drawn sketch photographed and inserted into your report is entirely acceptable. The diagram must show: where requests originate, what the server does, where rendering happens, and how data flows.

05
Stress-test your understanding.

Deliberately break something in one version and observe what fails. Note your observations. This kind of intentional investigation is what turns a lab exercise into genuine learning.

Tool freedom note: You can use Docker Compose to spin up an environment, deploy on a local virtual machine, use a cloud sandbox, or simply run processes on your localhost. If none of the tools mentioned feel right for your context, research alternatives and justify your choice in the report. Creating your own scaffolding scripts or tooling is warmly encouraged.

Module 4 · Application Types & Architectures

12.

Mobile Applications

≈ 1 page in your report

Native React Native Flutter Ionic Responsive PWA

Background and Why It Matters

Mobile traffic accounts for more than half of global internet usage, and that proportion continues to grow. As a web developer, you cannot afford to treat mobile as an afterthought — yet the decision of how to build for mobile is far from simple. The landscape ranges from fully native applications to purely web-based solutions, with a rich spectrum of hybrid approaches in between.

1
Phase 1 · Theory & Context

Native, hybrid, web — and the spectrum in between.

≈ 45 min

2.1Native Applications

Native applications are built using the platform's own language and SDK (Software Development Kit): Swift or Objective-C for iOS (Apple's mobile operating system), Kotlin or Java for Android (Google's mobile operating system). They have direct access to all device APIs — camera, GPS (Global Positioning System), accelerometer, Bluetooth — and generally deliver the best performance and the most consistent user experience.

The cost is development effort: maintaining two separate codebases for two platforms is expensive, time-consuming, and requires specialised expertise for each.

2.2Hybrid and Cross-Platform Approaches

Cross-platform frameworks attempt to write once and run on multiple platforms. React Native compiles JavaScript to native UI components. Flutter uses its own rendering engine and Dart language. Ionic wraps a web application in a native container using WebView. Each makes different trade-offs between developer experience, performance, and access to native capabilities.

Research each of these critically. None of them is universally superior. The right choice depends on the project's performance requirements, the team's existing skills, and the target audience's devices.

2.3Responsive Web Design vs. PWA vs. Native

Understanding the continuum is important:

Approach Offline? App Store? Native APIs? One Codebase?
Responsive WebPartialNoLimitedYes
PWAYes (service worker)SometimesGrowingYes
Hybrid (Ionic, etc.)YesYesMostYes
Cross-platform (RN, Flutter)YesYesYesMostly
NativeYesYesAllNo

Examine this table critically. Entries like “Sometimes” or “Growing” exist because the ecosystem is moving. Verify current capabilities yourself — what was true two years ago may no longer be accurate.

2.4Responsive Design Principles

Even if you are building a web application, it must function well on mobile screens. This means fluid grid layouts, flexible images, and breakpoints managed through CSS (Cascading Style Sheets) media queries. The MDN Web Docs on Responsive Design provide a reliable and continuously maintained reference.

Study the concept of a mobile-first approach: design and code for the smallest screen first, then progressively enhance for larger viewports.

Module 5 · Security & Testing

13.

Web & Mobile
Applications Testing

≈ 1.5 pages in your report

Unit Integration E2E TDD CI/CD Accessibility

Background and Why It Matters

Testing is, without exaggeration, one of the most undervalued skills in early-career software development — and one of the most valued by professional teams. An application with no tests is not finished: it is a liability. Any change, however small, can silently introduce regressions. Testing is how you build justified confidence in your code.

The landscape is layered. Different types of tests catch different categories of problems. Understanding what each layer does, and where it falls short, is more valuable than memorising any particular testing framework.

1
Phase 1 · Theory & Context

The pyramid, the discipline, the pipeline.

≈ 45 min

3.1The Testing Pyramid

The testing pyramid, popularised by Mike Cohn and discussed extensively in Martin Fowler's writing (see martinfowler.com/bliki/TestPyramid.html), describes an ideal distribution of test types:

Unit tests — the wide base
Test individual functions, methods, or components in isolation. Fast to run, cheap to write, easy to pinpoint failures.
Integration tests — the middle layer
Test how units collaborate — a controller calling a service that queries a database, for instance.
End-to-End tests (E2E) — the narrow top
Simulate real user behaviour across the full stack, including the browser. Powerful but slow and brittle.

The pyramid shape suggests where you should invest the most effort. Many teams invert this inadvertently — writing mostly E2E tests — which produces slow, fragile test suites.

3.2TDD (Test-Driven Development)

TDD (Test-Driven Development) is a discipline where you write a failing test before you write the code that makes it pass. The cycle — Red (failing test), Green (minimal passing code), Refactor (clean up) — sounds counterintuitive, but it produces code that is more modular, more focused, and more testable by design.

You are not required to adopt TDD for this lab, but you are required to understand it well enough to discuss it critically in your report.

3.3Testing Mobile Applications

Mobile testing adds dimensions beyond browser testing: device fragmentation, different operating system versions, touch interactions, geolocation, camera access, offline behaviour, and battery impact. Strategies include:

  • Device farms — services that run your tests on real devices. See BrowserStack or Sauce Labs — or research entirely different providers and tools.
  • Emulators and simulators — software that mimics device behaviour locally.
  • Accessibility testing — ensuring the application is usable by people with disabilities. WCAG (Web Content Accessibility Guidelines) from the W3C (World Wide Web Consortium) define the international standard.

3.4CI/CD (Continuous Integration and Continuous Deployment) and Automated Testing

Modern development teams do not run tests manually. Tests are triggered automatically whenever code is pushed to a repository — this is CI (Continuous Integration). The results determine whether the code can be merged and eventually deployed. Understanding this pipeline, even conceptually, is part of becoming a professional developer.

2
Phase 2 · Hands-on Lab

Write a meaningful test suite for your app.

≈ 1 h 15 min

Objective: Write and run a meaningful test suite for the application you have been building across Tasks 11 and 12.

01
Set up a testing framework.

Choose a testing tool appropriate for your application's technology stack. If you built with JavaScript, consider Jest, Vitest, Mocha, or any other tool you research and find suitable. If you used Python, consider pytest or unittest. There is no mandatory choice — your responsibility is to justify your selection.

02
Write at least five unit tests.

Identify five specific, testable behaviours in your application — pure functions, data transformations, component rendering logic, API response parsing. Write a test for each. At least two of them must initially fail (test-first approach), and you must then write the code to make them pass.

Record everything: the test code, the output before your fix, and the output after.

03

Test a flow that involves at least two components or layers working together. For example: a function that fetches data from a mock API and returns a transformed result, end-to-end within your code (not yet involving a real browser).

04

Use a browser automation tool to simulate a user visiting your application and completing an interaction. Playwright and Cypress are widely used, as is Selenium WebDriver — but you are free to use any tool you consider appropriate. Document the test scenario in plain English first (what the user does, step by step), then implement it.

05
Test on mobile.

Run your PWA on a real device or in device emulation mode and manually test at least three user journeys. Document each test: the scenario, the expected outcome, the actual outcome, and any discovered defect. If you find and fix a bug, document that too — this is valuable professional practice.

Important: The tools mentioned in this lab are well-established and widely documented, but they are not the only options. Research alternatives. Some of you may find tools that are better suited to your specific setup. Others may choose to build a simple custom test runner to understand how testing frameworks work internally. Both are valid approaches — as long as you document and justify them thoroughly.

Module 5 · Security & Testing

14.

Web Application
Security Basics

≈ 1 page in your report

OWASP Top 10 SQLi XSS CSRF AuthN/Z HTTPS CSP

Background and Why It Matters

Security is not a feature you add at the end of a project. It is a property that must be designed in from the first line of code. The overwhelming majority of security breaches are not the result of sophisticated state-sponsored attacks — they are the result of well-known, preventable vulnerabilities that developers simply did not address.

This task introduces you to the most critical and most common of those vulnerabilities, and to the professional mindset that takes security seriously from day one.

!

Critical Disclaimer for Task 14

The techniques you will explore in this task — including vulnerability identification, injection testing, and authentication analysis — are techniques used daily by professional security researchers, penetration testers, and security engineers. They are taught here for strictly educational purposes, and must be applied exclusively to applications and environments that you have built yourself or that have been explicitly provided for this lab.

Applying these techniques to any system, application, or network without explicit written authorisation from its owner is illegal in virtually every jurisdiction, regardless of intent. Your teacher takes no responsibility for any use of this knowledge outside the authorised educational context. Be a professional. Be ethical.

1
Phase 1 · Theory & Context

OWASP, injections, identity, transport.

≈ 45 min

4.1OWASP (Open Web Application Security Project) Top 10

The single most important reference document in web application security is the OWASP (Open Web Application Security Project) Top 10. It lists the ten most critical security risks to web applications, updated periodically based on data from real-world incidents and expert consensus.

Read it. All of it. It is freely available at https://owasp.org/www-project-top-ten/

The OWASP Foundation is a non-profit organisation dedicated to improving software security. Their resources are open, free, and authoritative.

4.2Injection Attacks: SQL Injection and Beyond

SQL (Structured Query Language) injection occurs when user-supplied data is incorporated into a database query without proper sanitisation, allowing an attacker to modify the query's logic. A classic example: a login form that checks SELECT * FROM users WHERE username = '[input]' AND password = '[input]' can be trivially bypassed by entering ' OR '1'='1 as the username.

-- Vulnerable: string concatenation in SQL
SELECT * FROM users
 WHERE username = '[input]'
   AND password = '[input]';

-- Attacker enters as username:
--   ' OR '1'='1
-- Resulting query is always true:
SELECT * FROM users
 WHERE username = '' OR '1'='1'
   AND password = 'whatever';

Beyond SQL, injection applies to operating system commands, XML (Extensible Markup Language) parsers, and any system that interprets user-supplied strings as instructions.

Mitigations: parameterised queries (also called prepared statements), ORM (Object-Relational Mapper) frameworks that enforce safe querying by default, and rigorous input validation.

// Safe: parameterised query (Node.js + pg)
const { rows } = await db.query(
  'SELECT id FROM users WHERE username = $1 AND password_hash = $2',
  [username, hashed]
);

4.3XSS (Cross-Site Scripting)

XSS (Cross-Site Scripting) allows an attacker to inject malicious JavaScript into a page viewed by other users. A stored XSS attack, for instance, might embed a script in a comment field that runs in every visitor's browser, stealing session cookies or redirecting to phishing pages.

Mitigations: output encoding (ensuring that user-supplied content is displayed as text, never interpreted as HTML or JavaScript), a strict Content Security Policy (CSP) header, and using templating engines that escape output by default.

4.4CSRF (Cross-Site Request Forgery)

CSRF (Cross-Site Request Forgery) tricks an authenticated user's browser into making an unauthorised request to a web application where the user is logged in. Because the browser automatically sends cookies with every request to the target domain, the server cannot distinguish the legitimate user from a forged request — unless the application specifically defends against it.

Mitigations: synchronised token patterns (a CSRF token embedded in forms and validated server-side), the SameSite cookie attribute, and verifying the Origin and Referer headers.

4.5Authentication and Session Management

Weak authentication is among the leading causes of account compromise. Common mistakes include: storing passwords in plain text, using weak hashing algorithms (MD5 or SHA-1 are not acceptable for passwords), never expiring sessions, and not rate-limiting login attempts.

Modern best practice: hash passwords with bcrypt, scrypt, or Argon2 (an algorithm specifically designed for password hashing that won the Password Hashing Competition). Use HTTPS (Hypertext Transfer Protocol Secure) everywhere. Implement MFA (Multi-Factor Authentication). Set session timeouts.

4.6HTTPS and TLS (Transport Layer Security)

Any data transmitted over plain HTTP is readable by anyone with access to the network path — a serious risk for passwords, personal information, and session identifiers. TLS (Transport Layer Security) encrypts the connection between the browser and the server. Today, there is no legitimate reason to serve a web application over plain HTTP.

For testing and local development, tools like mkcert allow you to generate locally trusted certificates. For production, Let's Encrypt provides free, automated, open TLS certificates.

4.7Security Headers

A set of HTTP response headers allow a server to instruct the browser to enforce security policies. Important ones include:

  • Content-Security-Policy — restricts sources from which scripts, styles, and other resources can be loaded
  • Strict-Transport-Security — forces HTTPS connections
  • X-Frame-Options — prevents clickjacking by controlling whether the page can be embedded in an iframe
  • X-Content-Type-Options — prevents MIME-type sniffing
Strict-Transport-Security: max-age=31536000; includeSubDomains
Content-Security-Policy: default-src 'self'; script-src 'self'
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin

Check your application's headers at securityheaders.com or any equivalent tool you find reliable.

2
Phase 2 · Hands-on Lab

Audit, exploit, mitigate — on systems you own.

≈ 1 h 15 min

Objective: Audit the security posture of your own application, identify vulnerabilities, and implement mitigations.

01
Set up an intentionally vulnerable test environment.

Do not test on your production application or on any live system. Create a local copy of your application, or use a purpose-built intentionally vulnerable environment. Well-known options include DVWA (Damn Vulnerable Web Application) or WebGoat — both maintained by the security community for exactly this educational purpose. But again: you are encouraged to research, compare, and choose the environment that best fits your learning objective.

02
Attempt a SQL injection on a form in your test environment.

Find a form in your test application that queries a database. Attempt basic SQL injection payloads. Document exactly what you submitted, what the application returned, and what that tells you about the underlying vulnerability. Then implement a parameterised query fix and verify that the injection no longer works.

03
Attempt a reflected XSS attack.

Find an input field whose value is reflected in the page's output. Submit a payload such as <script>alert('XSS')</script> and observe whether it executes. Document your findings. Then implement proper output encoding and verify the fix.

04
Review your authentication implementation.

Examine how your application handles user authentication (even if it is very simple). Answer the following honestly in your documentation: Are passwords stored safely? Are sessions invalidated on logout? Is there any rate limiting on login attempts? Is HTTPS enforced? Produce a short security checklist with a pass/fail for each item.

05
Add security headers.

Configure your server to send at least three of the security headers described in the theory section. Verify them using your browser's developer tools or an online header checker.

Note on tools: The payloads and techniques explored in Steps 2 and 3 are standard knowledge in the security field. Every responsible security practitioner knows them — because you cannot defend against an attack you do not understand. The tools mentioned (DVWA, WebGoat) are maintained specifically to make this type of learning safe and legal. Research alternatives if they do not fit your environment. Build your own vulnerable test application if you feel ambitious — it is one of the most educational things you can do.
